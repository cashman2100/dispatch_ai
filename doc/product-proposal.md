# Product Proposal: DispatchAI

## 1. Обоснование идеи

### Прикладная задача

Платформа «Мой Дом Онлайн» (МДО) обслуживает управляющие компании (УК) и ТСЖ, предоставляя CRM для обработки заявок жителей. Звонки принимаются двумя способами: живыми диспетчерами (вручную заносят в CRM) и голосовым роботом Aimylogic (автоматически создаёт заявку с транскрипцией и MP3).

Технически звонок проходит так: телефон УК → ВАТС оператора связи → SIP → ВАТС МДО (FreeSwitch/FusionPBX) → Aimylogic. ВАТС МДО сохраняет аудио-дорожку, робот передаёт транскрипцию и структурированные поля в CRM.

За 9 лет в CRM накоплены сотни тысяч аудиозаписей и транскрипций. Сейчас они используются только как приложение к заявке — для разбора споров.

### Почему это важно

Опытный диспетчер держит в голове знания, которых нет ни в одной базе данных: особенности конкретных домов (где старые трубы, какой лифт регулярно ломается), контакты и зоны ответственности сотрудников (сантехник Петров — дома на Ленина), историю проблем (кровля этого дома течёт каждую осень, капремонт в сентябре), контекст по жильцам (в кв. 45 пожилая женщина, повторное обращение).

Хороших диспетчеров мало, им нужно платить зарплату, а заменить их роботом «в лоб» нельзя — робот не знает контекста и воспринимается жителями как «тупой». При этом уволить диспетчера — значит потерять все его знания безвозвратно.

### Гипотеза

Если мы будем автоматически извлекать знания из речи диспетчеров и накапливать их в структурированной базе, то УК снижают зависимость от конкретных людей, а МДО получает основание для нового продукта — робота, который работает не по скрипту, а на базе знаний конкретной УК.

## 2. Цель и метрики

### Цель PoC

Создать pipeline, который берёт аудиозапись звонка, разделяет на роли (диспетчер/житель), транскрибирует каждую роль отдельно и извлекает из речи диспетчера структурированные факты. Факты сохраняются в базу с привязкой к адресу и доступны через CLI.

### Метрики успеха

**Продуктовые:**

| Метрика | Целевое значение | Как измеряем |
|---------|-----------------|-------------|
| Доля звонков, из которых извлечён хотя бы 1 факт | ≥ 40% | Автоматически по выходу pipeline |
| Точность извлечённых фактов | ≥ 80% precision | Ручная валидация выборки 100 фактов экспертом |

**Технические (NLP):**

| Метрика | Целевое значение | Как измеряем |
|---------|-----------------|-------------|
| DER (Diarization Error Rate) | ≤ 15% | Тестовый сет, 20–30 звонков с ручной разметкой ролей |
| WER (Word Error Rate) ASR | ≤ 25% | Тестовый сет, сравнение с ручной транскрипцией |
| F1 извлечения фактов | ≥ 0.65 | Размеченный тестовый сет, 200–300 фрагментов |

**Агентские:**

| Метрика | Целевое значение | Как измеряем |
|---------|-----------------|-------------|
| Доля pipeline-прогонов без ошибок | ≥ 90% | Логи |
| Среднее время обработки одного звонка | ≤ 5 мин | Замер end-to-end |

## 3. Сценарии использования

### Сценарий 1: Основной (happy path)

В папке `data/raw/` лежит MP3 звонка и CSV с метаданными (ID заявки, адрес дома).

1. Пользователь запускает `python -m dispatchai process call_0042.mp3`.
2. Модуль диаризации разделяет аудио на сегменты диспетчера и жителя.
3. Модуль ASR транскрибирует каждую роль отдельно.
4. Extraction Agent получает текст диспетчера и извлекает факты:
   - Диспетчер: *«Да, Мария Ивановна, у вас в прошлом году такое уже было. Я сантехника Петрова отправлю, он ваш дом знает. Капремонт у вас по плану в сентябре.»*
   - Извлечённые факты:
     - `{category: "жилфонд", fact: "протечка — повторяющаяся проблема, ежегодная"}`
     - `{category: "сотрудники", fact: "ответственный сантехник — Петров"}`
     - `{category: "жилфонд", fact: "капремонт запланирован на сентябрь"}`
5. Факты записываются в SQLite с привязкой к адресу и дате.
6. Пользователь запускает `python -m dispatchai search "Ленина 12"` и видит все накопленные факты по этому дому.

### Сценарий 2: Звонок без полезных фактов

Житель звонит, коротко описывает проблему, диспетчер принимает заявку по скрипту, не добавляя никакого контекста.

1. Pipeline отрабатывает, extraction agent возвращает пустой массив фактов.
2. В лог записывается: «звонок обработан, фактов не извлечено».
3. Это нормальный результат — не все звонки содержат уникальные знания.

### Сценарий 3: Edge-case — off-topic разговор

70% звонка — обсуждение погоды, личных дел, сплетней.

1. Extraction Agent через промпт проинструктирован извлекать только факты, относящиеся к жилфонду, сотрудникам и истории проблем.
2. Off-topic фрагменты игнорируются — агент возвращает факты только из релевантных частей (или пустой массив).

### Сценарий 4: Edge-case — несколько проблем в одном звонке

Жительница за один звонок сообщает: протечка в ванной, сломанный домофон, спрашивает когда покрасят подъезд.

1. Extraction Agent извлекает факты по каждой теме отдельно, каждый с отдельным тегом.
2. В SQLite сохраняются все факты, привязанные к одному звонку.

### Сценарий 5: Edge-case — диспетчер даёт неверную информацию

Диспетчер говорит «капремонт в следующем месяце», но это неверно.

1. На этапе PoC — факт сохраняется как есть, со ссылкой на источник (ID звонка, timestamp).
2. При ручном просмотре через CLI — пользователь видит источник и может оценить достоверность.
3. Система не претендует на верификацию фактов — это out-of-scope для PoC.

### Сценарий 6: Edge-case — плохое качество аудио

Фоновый шум, диспетчер и житель говорят одновременно, связь прерывается.

1. Диаризация может ошибиться (перепутать роли).
2. ASR может дать высокий WER.
3. Extraction Agent получает «мусорный» текст и либо извлечёт мало фактов, либо нет.
4. В лог записывается предупреждение, если confidence диаризации ниже порога.

## 4. Ограничения

### Технические

| Параметр | Значение | Обоснование |
|----------|---------|------------|
| Режим работы | Offline, batch | Не вмешиваемся в production |
| Макс. длина аудио | 30 мин | 95% звонков < 10 мин |
| Формат входа | MP3/WAV + CSV с метаданными | Формат экспорта из CRM МДО |
| Язык | Русский | Целевой рынок |
| Выход | JSON → SQLite | Простейшее хранилище |

### Операционные

| Параметр | Значение |
|----------|---------|
| Бюджет на LLM API | ~$50/мес (PoC) |
| GPU | Google Colab / 1 облачная GPU для Whisper + pyannote |
| Данные | 100–500 звонков с MP3 из CRM МДО (экспорт) |
| Разметка для NLP-курса | 200–300 фрагментов вручную |

## 5. Архитектурный набросок

### Модули

| Модуль | Вход | Выход | Технология |
|--------|------|-------|-----------|
| **Diarization** | MP3 | Список сегментов `[{start, end, speaker}]` | pyannote.audio 3.x |
| **ASR** | Аудио-сегменты по ролям | Текст диспетчера, текст жителя | Whisper large-v3 |
| **Extraction Agent** | Текст диспетчера + адрес дома | JSON с фактами по таксономии | LLM (Claude/GPT) |
| **Knowledge Store** | JSON-факты | Запись в SQLite, поиск по адресу | SQLite + Python |
| **Orchestrator** | MP3 + метаданные | Вызов модулей последовательно, логирование | Python-скрипт |
| **CLI** | Команда пользователя | Результат в терминале | Click/Typer |

### Таксономия фактов (для Extraction Agent)

```json
{
  "fact_id": "auto-generated",
  "call_id": "call_0042",
  "address": "ул. Ленина, д. 12",
  "timestamp": "2025-03-01T14:30:00",
  "category": "жилфонд | сотрудники | история_проблем",
  "subject": "кровля",
  "fact_text": "Протечка повторяется ежегодно осенью",
  "source_quote": "у вас в прошлом году такое уже было",
  "confidence": 0.9
}
```

### Что делает LLM, а что — обычный код

| Этап | LLM | Обычный код |
|------|-----|------------|
| Получение MP3 из папки | | ✅ |
| Диаризация | | ✅ (pyannote — ML-модель, но не LLM) |
| ASR | | ✅ (Whisper — ML-модель, но не LLM) |
| Извлечение фактов из текста диспетчера | ✅ | |
| Запись в SQLite | | ✅ |
| Поиск по адресу | | ✅ (SQL-запрос) |

## 6. Data Flow

```
data/raw/
  ├── call_0042.mp3
  └── metadata.csv  (call_id, address, date)
          │
          ▼
   ┌──────────────┐
   │ Orchestrator  │
   └──────┬───────┘
          │
          ▼
   ┌──────────────┐      segments[]
   │  pyannote    │─────▶ {start: 0.0, end: 5.2, speaker: "A"}
   │  diarization │      {start: 5.2, end: 12.1, speaker: "B"}
   └──────────────┘      ...
          │
          ▼  (определяем: speaker A = диспетчер, B = житель)
   ┌──────────────┐
   │   Whisper    │─────▶ dispatcher_text, resident_text
   │   ASR        │
   └──────────────┘
          │
          ▼
   ┌──────────────┐
   │  LLM Agent   │─────▶ facts[] (JSON по таксономии)
   │  extraction   │
   └──────────────┘
          │
          ▼
   ┌──────────────┐
   │   SQLite     │      facts table:
   │   store      │      call_id | address | category | fact_text | ...
   └──────────────┘
          │
          ▼
   ┌──────────────┐
   │    CLI       │      $ dispatchai search "Ленина 12"
   └──────────────┘      → 7 фактов найдено
```
